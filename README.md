# WRF-downscaled ERA5 Daily Zonal Stats

This project computes daily zonal statistics for polygons from curated WRF-downscaled ERA5 4km data. The pipeline is variable-agnostic: which polygons, which variables to process, how each is aggregated, and what the output columns are named are all driven by config, not hardcoded. The shipped config here (`config/pipeline_config.json`) computes the daily zonal min, mean, and max of the `t2_mean` (daily mean air temperature) field for a select set of MERIT catchments, but any other curated variable(s) and other polygons can be substituted without code changes (see [Config-Driven Parameters](#config-driven-parameters)).

Primary runtime configuration lives in `config/pipeline_config.json`.

## Quick Start:

Run from the repo root (`wrf-downscaled-era5-zonal-stats/`):

```bash
# Create and activate the environment
micromamba create -y -f environment.yml
micromamba activate wrf-era5-zonal-stats
```

```bash
# Run with slurm (uses config/pipeline_config.json by default)
```

# Processing Pipeline Details

- Catchments: shapefile path set via `io.shapefile` in config, keyed on the field named in `io.id_field` (default `COMID`)
- Curated variable root: path set via `io.data_root` in config, containing one subdirectory per variable named in `variables[].variable`, e.g. for the shipped config (all three entries read the same `t2_mean` variable, see "What the pipeline computes" below):
  - `t2_mean/t2_mean_<YEAR>_daily_era5_4km_3338.nc`

## Outputs
- Final table: path set via `outputs.final_path` in config (e.g. `outputs/zonal_t2_daily_<START>_<END>.csv` for the shipped config)
- Year partitions: path set via `outputs.yearly_dir` in config (e.g. `outputs/zonal_t2_daily/year=YYYY/part.csv`)
- QC histograms: PNG with one histogram per variable, written next to the final table with the same base filename (e.g. `outputs/zonal_t2_daily_<START>_<END>.png`); generated by `scripts/finalize_zonal.py`
- QC summary: path set via `outputs.qc_path` in config (e.g. `outputs/qc_summary.json`)
- Preflight summary: path set via `outputs.preflight_summary` in config (e.g. `outputs/preflight_summary.json`)

## What the pipeline computes
Per catchment ID per date, one column per entry in `variables` plus a cell count. With the shipped config (all three entries read the `t2_mean` variable; only the `aggregation_method` differs):
- `cell_count` (intersecting cell count â€” see "Zonal Engines" below for exactly what this means under the default `exactextract` engine)
- `t2_mean_zonal_min`: zonal minimum of `t2_mean`
- `t2_mean_zonal_mean`: zonal mean of `t2_mean`
- `t2_mean_zonal_max`: zonal maximum of `t2_mean`

## Zonal Method and Intersecting Cells

This section describes the `rasterize` engine specifically (see "Zonal Engines" below for the `exactextract` engine, which the shipped config actually uses by default).

- The `rasterize` engine rasterizes catchment polygons to the source grid in the CRS set by `spatial.target_crs` (default EPSG:3338).
- Cell membership mode is controlled by `spatial.cell_membership` in config (`all_touched` or `center`); both Slurm launchers inherit this setting. This option only applies to the `rasterize` engine; it is ignored, with a printed notice, under the `exactextract` engine.
- In `all_touched` mode, a grid cell is counted as intersecting if any part of that cell is touched by the catchment polygon. In `center` mode, a cell is counted only if its center point falls within the catchment polygon.
- Under this engine, `cell_count` is fixed per catchment because this geometric lookup is created once and reused for all dates.
- For each entry in `variables`, the configured `aggregation_method` (`min`, `mean`, or `max`) is applied daily over each catchment's member cells, producing the named `output` column.

## Zonal Engines: `rasterize` vs `exactextract`

This pipeline supports two interchangeable zonal-stats engines, selected via `spatial.engine` in config or `--engine` on the command line. The code defaults to `"rasterize"` if `spatial.engine` is omitted entirely, but **the shipped `config/pipeline_config.json` explicitly sets `spatial.engine: "exactextract"`**, so that is what actually runs by default in this repo. Both engines produce the same output schema (`<id_field>, date, cell_count` + one column per configured variable).

- **`rasterize`**: rasterizes catchment polygons onto the source grid once, using binary `all_touched`/`center` cell inclusion (see "Zonal Method and Intersecting Cells" above), then averages/min/maxes the included cells per day with plain NumPy. `cell_count` is fixed per catchment, computed once from geometry alone.
- **`exactextract`** (what the shipped config uses): uses the [exactextract](https://isciences.github.io/exactextract/) library to compute true pixel-coverage-fraction overlays between each polygon and the raster grid, instead of a binary touched/not-touched rule per cell.
  - `mean`-type variables are **coverage-fraction-weighted** (a cell that's 30% inside the polygon contributes 30% of its weight to the average) â€” a real accuracy improvement over `rasterize`'s simple average of whichever cells happen to be flagged as included.
  - `min`/`max`-type variables are **unweighted** (any cell touching the polygon at all is eligible, matching the spirit of `rasterize`'s `all_touched` mode).
  - Unlike `rasterize`, each polygon's coverage is computed independently, so overlapping or nested catchments are not mutually exclusive â€” there is no "last polygon drawn wins the pixel" effect. (The 122 catchments shipped in `shp/merit_watersheds/` were checked during development and are *not* meaningfully overlapping, so this mainly matters if different/nested polygon sets are used in the future.)

### `cell_count` means the same thing under both engines, but is computed differently

Under both engines, `cell_count` is a literal tally of grid cells that intersect the polygon at all (any nonzero overlap counts as a full cell, regardless of how much of the cell is actually covered) â€” conceptually the "how many cells went into this calculation" number a non-technical reader would expect.

The difference is **when** it's computed:
- `rasterize` computes it once, up front, from geometry alone. It is therefore fixed forever for a given polygon, regardless of which day you look at.
- `exactextract` recomputes it **per day**, counting only that day's non-NaN cells (using exactextract's `count(coverage_weight=none)` operation). **If you ever see `cell_count` change from one date to the next for the same polygon under the `exactextract` engine, that means one or more of that polygon's intersecting cells had missing (NaN) source data on that specific day â€” it is not a bug, and the polygon's geometry has not changed.** In practice, for the `t2_min`/`t2_mean`/`t2_max` source data available to this pipeline, this was checked for 2000, 2005, and 2023 and the NaN pattern is identical across all three temperature variables and constant across every day of the year (it marks fixed out-of-domain grid padding, not missing-data gaps) â€” so day-to-day variation in `cell_count` has not actually been observed in this dataset, even though the engine is capable of it.

Measured directly with the shipped config (year 2000, all 122 catchments, `t2_mean` source variable, both engines run back-to-back on the same grid/shapefile): `cell_count` matched exactly for 28,548 of 44,652 polygon-days (64%); where it differed, the mean absolute difference was about 1.3 cells (max 9). This is **not** caused by nested/overlapping catchments (confirmed not applicable here) â€” it comes from `rasterize`'s GDAL-based `all_touched` scanline algorithm and exactextract's exact vector/raster overlay disagreeing at a small number of boundary pixels per catchment. Expect the two engines' cell counts to be close, not bit-identical.

### Stat values will differ slightly between engines

This is expected, not a bug â€” it's the direct consequence of coverage-fraction weighting. Measured with the shipped config (year 2000, `t2_mean` source variable, both engines):

`mean` is well-behaved because coverage weighting smooths out boundary disagreements. `min`/`max` have a longer tail of larger outliers because they pick a single extreme cell's value â€” if the two engines disagree on whether one boundary cell with an unusually high or low value counts at all, the catchment's min/max can swing noticeably even though the average barely moves.

### Performance

Measured directly on this pipeline, one full year (2000, 366 days), shipped config (`t2_mean` aggregated three ways), 122 catchments, single-threaded, no Slurm:

| Engine | Wall time |
|---|---|
| `rasterize` | 7m 33s |
| `exactextract` | well under 30s |

`exactextract` is roughly **15-20x faster** here, mainly because it reads each NetCDF chunk once and matches all overlapping polygons against it (`strategy="raster-sequential"`), whereas the per-day NumPy loop in the `rasterize` engine re-reads/re-decompresses small source chunks repeatedly. Extrapolated to the full 2000â€“2023 run, `rasterize` is why this pipeline's Slurm setup parallelizes by year in the first place (~3 hours single-threaded, ~8 minutes per year in parallel); `exactextract` would complete the entire 24-year run in well under 10 minutes even running serially. The existing per-year Slurm array structure works unchanged with either engine and was kept as-is when `exactextract` was added, since per-year parallelism is still useful for observability/restart granularity even though it's no longer required for performance.

## Config-Driven Parameters
Edit `config/pipeline_config.json` to swap variables, datasets, or polygons without code changes:
- `io.shapefile`, `io.id_field`, `io.data_root`
- `years.start`, `years.end`
- `variables`: list of `{variable, aggregation_method, output}` entries â€” add, remove, or repurpose entries to process any variable present under `io.data_root`, not just temperature
- `naming.file_template`: e.g. `{variable}_{year}_daily_era5_4km_3338.nc`
- `spatial.target_crs`, `spatial.cell_membership` (rasterize engine only)
- `spatial.engine`: `"rasterize"` or `"exactextract"` (code default if omitted is `"rasterize"`; the shipped config sets `"exactextract"` explicitly) â€” see "Zonal Engines" above
- `outputs.*` for preflight/yearly/final/qc paths and format
- `performance.time_chunk`, `performance.workers_default`

## Scripts
- `scripts/check_inputs.py`: preflight validation of shapefile and per-variable input files
- `scripts/zonal_watersheds.py`: main per-year zonal computation, finalize, and QC; dispatches to either zonal engine based on `spatial.engine`
- `scripts/exactextract_zonal.py`: `exactextract`-engine implementation, used by `zonal_watersheds.py` when `spatial.engine` is `"exactextract"`
- `scripts/finalize_zonal.py`: standalone merge of yearly outputs and QC (used by the Slurm array + finalize workflow)
- `scripts/config_utils.py`: shared config loading helpers
